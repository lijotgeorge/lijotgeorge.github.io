Etch-a-Sketch

2021-09-07

Hooo boy! That took a while!

Last I left off was when I had made my first webpage as Google homepage replica. After that was JavaScript. As someone with experience in programming in Python as well as C to some extent I felt confident I would be able to handle JS pretty well too. And I did… at least in the beginning.

The initial lessons that TOP set up teaches you the basics of programming in JS. This means variable declarations, arrays, conditional statements and functions. It was a little confusing learning the syntax for a new language again but turns out JS has a lot of similarity (at least in syntax) with C so falling back into rhythm didn’t take long. The biggest stumbling block was getting used to the console. In C and Python I’m used to typing everything in the terminal and immediately figuring out what I did and where I’m going wrong. It wasn’t so straightforward in JS. The interplay between what I want the webpage to do and what JS is doing is a bit harder to figure out when you need to switch windows every damn time!

And that’s even before you learn just how much you can control the webpage simply with JavaScript!

Welcome to the Document Object Model (DOM).

The DOM, I’ve come to learn, is the backbone behind JS’s ability to manipulate webpages. Essentially, DOM is a node-like structure of any html webpage as defined in the the html document. This then allows JS to manipulate the entire html document and add, remove or modify the various elements in the webpage and along with CSS make stuff look/happen any which way you want.

This leads nicely to the first project in the JavaScript course: Etch-a-Sketch.

Before I talk about the challenge of creating this app let me just plug in my finished product. Demo of the project.

The Process

As I mentioned before this challenge had me make extensive use of the DOM and while the challenge page did mention that you would need to search for a lot of new stuff there weren’t any specific suggestions as to where to look. The JS pages on W3Schools while good are hardly comprehensive or very explanatory.

After searching through google and youtube I came across a playlist by The Net Ninja that I found quite nice. This still wasn’t enough though and I still had to google for some specific issues as and when they arose. But that’s the whole point of this I suppose.

(So it turns out I’m an idiot! I completely skipped over the section of the course that dealt with the DOM and how to use JS to handle it ü§¶)

I’m not going to go into the nitty-gritty of the application but I do want to write down some interesting thoughts about the whole process. One of the initial lessons in the course was about ‘‘problem solving’' and ‘‘thinking like a programmer’'. A workflow of sorts was suggested that programmers should follow when attempting any problem. This included creating a pseudo-code of the problem and using the Divide and Conquer method when breaking down the possible solutions to the problem.

I did not do any of that for this problem!

At least not in the way it was suggested. Usually whenever I tackle a programming challenge I try to break down the problem into priorities – which subproblem is absolutely essential to this problem and which ones can be tackled later on. That’s what I did with this one as well.

Going in, the absolute lowest level solution was to have a grid divided into a fixed number of subgrids that change to black when I mouseover it. Once that was done the next step would be to figure out a way to delete any colored grid. Then figure out a way to randomize colors on every mouse over etc. etc. This would go on until the final solution is exactly as described in the problem. I also took inspiration from a few of the examples showcased in the project page.

But it turned out a whole lot messier than I had expected. With every rising complexity I had to rewrite some or most of the code to functionally include the feature into my solution. I wonder now if this is actually an inescapable feature of the process of creating web pages or could this have been avoided had I actually sat down and written what I wanted my final code to do right at the very beginning?

I don’t know the answer to that just yet but I’m going to try doing it this way for the next problem.
